When I first started learning C programming, I approached it the same way many beginners do.

Write a program.

Compile.

Fix errors.

Move on to the next program.

I was finishing exercises, but I wasn't really thinking about **how software is built**.

This week, that changed.

Instead of treating every C program as a standalone assignment, I started treating each one like a small software project using **GitHub Issues**.

It completely changed how I approach programming.

### From "Finish the Program" to "Finish the Feature"

The project I'm currently building is a **Library Management System** in C.

At first, I thought of it as one big task.

Then I asked myself:

> *How would a software team build this?*

They probably wouldn't tell one developer:

> "Go build the entire library management system."

Instead, they'd break it into smaller features.

So I did the same.

My project now has issues like:

*   Create the `main()` function
    
*   Design the menu display
    
*   Implement `add_book()`
    
*   Implement `view_book()`
    
*   Learn and use structures
    
*   Register members
    
*   Issue books
    
*   Return books
    

Each issue became a small goal with its own acceptance criteria.

Instead of feeling overwhelmed by a large project, I only needed to focus on solving **one problem at a time**.

### Functions Started Making More Sense

Breaking the project into GitHub Issues also changed how I design my code.

Earlier, I would think:

> "I need to write the program."

Now I think:

> "What responsibility should this function have?"

For example:

*   One function displays the menu.
    
*   Another adds a book.
    
*   Another views a book.
    
*   Another validates member information.
    

Each function exists for a single purpose.

That way of thinking feels much closer to how embedded software is designed, where every module has a well-defined responsibility.

### Learning GitHub While Learning C

One unexpected benefit was that I wasn't just learning C anymore.

I was also learning how professional developers manage their work.

For every issue, I:

*   Create a dedicated feature branch.
    
*   Work only on that feature.
    
*   Commit the completed work.
    
*   Open a Pull Request.
    
*   Merge it back into `main`.
    

Even though I'm working alone, following this workflow has made the project feel much more organized.

It also gives me confidence that I'm building habits I'll use when working with a team in the future.

### AI Didn't Build It for Me

Another thing I noticed this week is how my relationship with AI continues to evolve.

Instead of asking:

> "Write the `add_book()` function."

I found myself asking questions like:

*   Does this function have a single responsibility?
    
*   Is my logic correct?
    
*   Am I thinking about the problem the right way?
    
*   What edge cases have I missed?
    

The difference is subtle but important.

I'm no longer trying to get to the answer as quickly as possible.

I'm trying to understand **why** the solution works.

That mindset has helped me think through problems instead of simply copying solutions.

### Thinking Like an Embedded Engineer

One thing I enjoy about this approach is that it reminds me of how embedded systems are built.

Firmware isn't usually written all at once.

It's built feature by feature.

One module controls communication.

Another handles sensors.

Another manages memory.

Another controls hardware.

By breaking my C projects into GitHub Issues, I feel like I'm practicing the same style of thinking—just on a much smaller scale.

### The Journey Continues

The Library Management System is still a work in progress.

There are still several issues left to complete, and I'm intentionally taking them one at a time.

Every completed issue teaches me something new—not just about C programming, but about software design, version control, and problem solving.

And that's exactly the kind of engineer I'm trying to become.

One feature at a time.

One commit at a time.

One lesson at a time.

### 🚀 Jarvis Learning Log

This week, Jarvis didn't learn a new algorithm.

Instead, he learned **how software gets built**.

✅ Breaking projects into GitHub Issues  
✅ Designing functions with a single responsibility  
✅ Using feature branches and Pull Requests  
✅ Thinking about software architecture before writing code  
✅ Using AI as a mentor instead of a solution provider

The Library Management System isn't finished yet—but the process of building it is teaching me far more than simply completing another C program.
